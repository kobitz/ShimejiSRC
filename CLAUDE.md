## Project Overview

A heavily modified fork of Shimeji-ee (Java/Windows desktop mascot app) with an AI assistant layer.

**North-star goal (Ko's words):** *a sci-fi-esque desktop companion you'd expect to only see in movies or anime* ‚Äî a mascot that genuinely perceives and inhabits your real digital environment, not a scripted toy. Every feature is a means to that end. Concretely: character-accurate personas that perceive and react to the user's environment in real time ‚Äî proactive behavior, environment awareness (windows, audio, screen, hardware, files), peer interactions, persistent memory. Always fully local (Ollama; no cloud/paid APIs). When weighing a design call, the tiebreaker is "does this make it feel more like that movie/anime companion."

## Architecture

**Behavior ‚Üí Action ‚Üí Animation ‚Üí Environment** (40ms tick loop)

- **Behavior** (`behavior/`): Long-term decision-making. `UserBehavior` is the standard implementation.
- **Action** (`action/`): Short-term execution, 41 types, all extend `ActionBase`.
- **`Manager`**: Tick loop. `WindowsEnvironment.beginTick()` runs once per tick (not per mascot). **Tick watchdog (June 2026):** any tick ‚â•250ms (120 ‚Üí 200 ‚Üí 250; raised to filter the large-population/envScan blips and surface only the huge spikes ‚Äî the gemma4 generation-starvation events are 400ms‚Äì1s) logs `[TickWatch] Slow tick:` WARNING with phase breakdown ‚Äî envScan ms, slowest mascot (image set + ms), mascot count, heap used/total, cpu/gpu load. Unattributed time (total ‚àí envScan ‚àí mascots) = lock wait (EDT holding the mascots lock) or GC. Added to diagnose intermittent ~1s drops to 3-4 fps; check ShimejieeLog for these lines after a stutter.
- **`Mascot`**: Individual agent (~3500 lines). `userData` HashMap for per-mascot cross-tick state.
- **`NativeFactory`**: Abstract factory; `Platform` selects Windows/Mac/Generic/Virtual impl.
- `WindowsEnvironment` (src_win): JNA ‚Üí Win32, LRU cache max 256. `MacEnvironment`: JNA ‚Üí Carbon/Cocoa.

### Configuration System

XML-driven: `conf/actions.xml` + `conf/behaviors.xml` (per-mascot overrides in `img/[name]/conf/`). Supports English and Japanese tag names. `${CONSTANT}` substitution.

- **`<AnimationTemplate>`**: Reusable frame list. Actions reference with `<Animation Template="X" ImageAnchor="48,128" Velocity="-2,0" />` ‚Äî per-action attributes override template values. Used by Paimon (~40 float variants from one template). Schema keys: `AnimationTemplate`, `Template`.

### Scripting

XML conditions/durations/velocities accept Nashorn JS expressions. **`ScriptFilter`** sandboxes ‚Äî no `java.lang`, no I/O. `VariableMap` exposes `mascot.*` and `environment.*`. Nashorn exposes Java bean getters as properties (`o.getImageSet()` ‚Üí `o.imageSet`). `Script.get()` caches `CompiledScript` per thread keyed by source (engines are per-thread `ThreadLocal`s, so the cache must be too) ‚Äî previously every `#{...}` evaluation recompiled its source each frame.

### Notable Engine Features

- **Hotkey System** (`HotkeyManager`): JNativeHook. **TWO sources, per-mascot overrides global per combo** (June 24 2026): (1) global `conf/hotkeys.properties` ‚Äî bare `Behavior` fires for all active mascots, `ImageSet:Behavior` targets one; (2) per-mascot `img/<ImageSet>/conf/hotkeys.properties` ‚Äî **bare** behavior names, implicitly scoped to that image set, **overriding** the global binding for the same combo (combos the per-mascot file doesn't define still fall through to global). `loadAllBindings()` reads the global file then scans `img/*/conf/hotkeys.properties`. Resolution is per (combo, imageSet) via `effectiveBindings()` ‚Äî returns the per-mascot bindings if that image set defines the combo, else the global parts whose scope includes it; **every fire/hold/cancel path goes through it** (`firePress` iterates `activeImageSets()`; `getHeldBehaviorFor` and `cancelHeldForCombo` resolve per-imageSet). Line format `combo=Behavior`; combo = `ctrl/shift/alt/meta`+key (`F1`/`A`/`SPACE`/`MOUSE1`-`5`). `!hold` suffix for hold-to-loop.
- **Affordance System**: `AffordanceStay` broadcasts a tag; `Manager.getNearestAffordance()` finds nearby mascots with that tag.
- **Scale Action**: Runtime resizing, scriptable `Target`. Batch pre-warms all `scalables` on scale step. `renderCX`/`renderCY` derived from source dimensions to stay pinned to anchor during async cache lag.
- **Two scaling stages (NOT redundant ‚Äî keep separate)**: **global `Scaling`** (Settings menu, `Mascot.getGlobalScaling()`) is the master load-time resolution ‚Äî `ImagePairLoader` re-rasterizes each sprite *from the source PNG* at this scale (HQX/bicubic capable), it's the `ImagePairs` cache-key resolution, and it bakes movement/velocity deltas at build (`AnimationBuilder`). Uniform, static, changing it forces `ImagePairs.clear()` + reload. **dynamic `currentScale`** (per-mascot `Scale.<imageSet>`, Scale action) is a live, animatable per-mascot multiplier applied at *render* time by `ScalableNativeImage` (async worker, LRU-16). They compose into one pipeline: displayed size & anchor = `native √ó global √ó currentScale`; movement = `rawMove √ó global √ó currentScale`. Each exists because the other can't do its job (global can't re-rasterize per-tick; dynamic must be cheap/animatable). **Both static `Scaling` caches (`Mascot.cachedGlobalScaling`, `ActionBase.scalingConstant`) are invalidated in `Main`'s settings image-reload path** (June 19 2026 fix ‚Äî v1.4 ‚Äî otherwise a live global-scale change kept the old anchor scale ‚Üí floor-clipping/turn-teleport until restart).
  - **Re-rasterize-from-source on dynamic scale** (June 19 2026): `ScalableNativeImage` (only active when `currentScale ‚âÝ 1`) draws from the **full-resolution source PNG**, not the already-globalScaling-shrunk bitmap ‚Äî so global `0.5` + dynamic `1.4` (effective `0.7`) rasterizes `1.0 ‚Üí 0.7` in one step instead of `1.0 ‚Üí 0.5 ‚Üí 0.7` (no double resample / compounded blur). Target dimensions are still `round(globalBakedSource.w √ó currentScale)`, so the final size ‚Äî and the `renderCX`/`renderCY` anchor math (`imageWidth / lastBaseImageWidth`) ‚Äî is byte-identical to the old path; only the *pixels* are sourced at higher quality. The native source (premultiplied by `Opacity`, mirrored for right-facing frames via `ImagePairLoader.premultiply`/`flip`, now package-visible) is loaded **once per frame on the worker thread** and cached on the instance (`MascotImage.sourcePath`/`isSourceFlipped` carry the on-disk path); falls back to resampling the baked bitmap when there's no PNG source (programmatic images) or a read fails. `currentScale ‚âà 1` is unchanged ‚Äî it uses the shared global-baked `ImagePairs` bitmap (already a single rasterization).
- **Audio Level in XML**: `mascot.environment.audioLevel` exposes real-time system speaker RMS (~0-32767).
- **Process Priority Separation** (`environment/ProcessPriorityUtil.java`, June 2026): TickWatch proved the intermittent 2-4 fps dips were CPU starvation during inference bursts (chat.log reaction at 22:36:37 = TickWatch cluster at the same second, cpu 97-99%; all tick phases inflated uniformly, heap healthy ‚Äî no app defect). Fix is rank, not rate: Shimeji raises itself to ABOVE_NORMAL at startup (`Main.run()`); the Whisper python server is demoted to BELOW_NORMAL at spawn (`WhisperProcess`); `CpuTempMonitor`'s fast loop demotes `ollama.exe`/`llama-server.exe` every 5s via Toolhelp32 scan (already-demoted skipped; BELOW_NORMAL parents spawn BELOW_NORMAL children, so demoting ollama.exe covers future runners). Custom JNA kernel32 mappings ‚Äî only core jna.jar is bundled, no jna-platform. GPU-contention dips (gpu 97% during generation) are NOT fixed by this ‚Äî priority is CPU scheduling only.
- **Hardware Sensors** (`CpuTempMonitor`): `cpuLoad`, `gpuTemp`/`gpuLoad`/`gpuMemFree`/`gpuMemTotal` (one persistent `nvidia-smi -l 1` stream, NVIDIA only ‚Äî replaced the old spawn-per-second model; OllamaClient reads the cached mem values for GPU fit), `batteryLevel` (JNA), `ramLoad`. `TempSensor.exe` streams `cpuTemp=XX.X` every 1s (LHM, requires admin).
- **Video Area Server** (`VideoAreaServer`): HTTP on localhost:41221. Browser extensions send video bounding boxes; mascots treat them as windows.
- **Tint Action** (`action/Tint.java`): Alpha-blended color overlay composited per-frame. `Target` (opacity) and `Color` accept script expressions (`#{...}`) ‚Äî stored on the mascot via `setTintExpr()`; `Mascot.tick()` re-evaluates both every 40ms regardless of what behavior is running. **`Target` is only processed when it is a script expression** ‚Äî for a static opacity use `Opacity` (e.g. `Opacity="0.5"`); `Target="0.5"` as a literal silently falls through to `Opacity` (defaults 0.0 if absent). Color uses lerped R/G/B channels for smooth sensor-driven shifts. `ClearOnExpiry="true"` snap-clears on duration expiry. Schema keys: `Color`, `Opacity`, `Target`, `LerpFactor`, `ClearOnExpiry`.
  ```xml
  <Action Name="TintWithLoad" Type="Embedded" Class="com.group_finity.mascot.action.Tint"
      Color="#{(function(){ var g=Math.max(0,Math.min(255,Math.round(mascot.environment.gpuLoad/100*255))); var h=('0'+g.toString(16)).slice(-2); return 'FF'+h+h; })()}"
      Target="#{(mascot.environment.ramLoad + mascot.environment.gpuLoad - 50) / 150}"
      LerpFactor="0.08" />
  ```
- **Transition Animations / Tweens** (`config/Configuration.java` + `behavior/TransitionBehavior.java` + splice in `config/ActionBuilder.java`, June 2026): a per-mascot `<TransitionList>` of `<Transition>` rules that slip a short tween between two states. Each rule names comma-separated `Before` and `After` **names**, with `Bidirectional="true"` also matching the reverse (one rule = e.g. Stand‚ÜîSit). Tween frames come from inline `<Animation>` children (built as an `Animate` ‚Äî leftover attributes like `BorderType`/`Draggable` become Animate params) OR a named `Action="..."`. **One registry, applied at TWO boundaries:**
  - **Behavior boundary** (matches *behavior* names): intercepts at the **weighted-random pick in `buildNextBehavior`** ‚Äî the chokepoint natural transitions flow through (covers pool selection *and* `NextBehaviorList` successors like Fall‚ÜíStand), deliberately **excluding** drag/throw/hotspot/hotkey/AI `setBehavior` (those use `buildBehavior`, not `buildNextBehavior`). A `TransitionBehavior` plays the tween then reaches the target via `buildBehavior` (NOT `buildNextBehavior`), so a tween can't recurse into another tween; it is intentionally **not** reported by `getCurrentBehaviorName()`, so tweens stay out of the behavior history/tooltip. Drag/throw still interrupt it.
  - **Action boundary** (matches *action-reference* names): for compound `Sequence`s that chain multiple actions in ONE behavior (e.g. 2B's `WalkLeftAlongFloorAndSit` = Walk‚ÜíStand‚ÜíLook‚ÜíStand‚ÜíSit). The behavior hook can't reach inside a sequence, so `ActionBuilder.spliceActionTransitions()` inserts the tween as an extra child step between consecutive `<ActionReference>`s whose names match a rule (built lazily at `buildAction` time; the built `Action[]` is parallel to the ref list, so names are recoverable via `Configuration.buildActionTween()`). **Sequence only**, never `Select` (Select children are alternatives, not a chain).
  
  Because behaviors and actions usually have distinct names (behavior `StandUp`/`SitDown` vs action `Stand`/`Sit`/`Walk`), a rule's names naturally select which boundary it fires at ‚Äî no cross-contamination. Used for short 1-2 frame in-betweens (the common case) and the rare detailed turn-around. **NOTE: a detailed turn-around does NOT need this system** ‚Äî `Move` already plays a `Turn="true"`-flagged animation on a facing flip (`Move.java`/`Animation.isTurn()`); "complex" there is just more frames. Schema keys (base `schema.properties` only; en/ja inherit via parent fallback): `TransitionList`, `Transition`, `Before`, `After`, `Bidirectional`. 2B uses two rules (both a single `shime30.png` frame): behavior-level `StandUp`‚Üî`SitDown`, and action-level `Walk,Run,Stand`‚Üî`Sit` (catches the compound walk-and-sit sequences).
  ```xml
  <TransitionList>
      <Transition Before="StandUp" After="SitDown" Bidirectional="true" BorderType="Floor">           <!-- behavior boundary -->
          <Animation><Pose Image="/shime30.png" ImageAnchor="64,112" Velocity="0,0" Duration="6" /></Animation>
      </Transition>
      <Transition Before="Walk,Run,Stand" After="Sit" Bidirectional="true" BorderType="Floor">         <!-- action boundary (inside Sequences) -->
          <Animation><Pose Image="/shime30.png" ImageAnchor="64,112" Velocity="0,0" Duration="6" /></Animation>
      </Transition>
  </TransitionList>
  ```
- **Direction-change / sudden-stop animations (worked example: Mario's dash-brake skid, June 2026)**: a reusable *config-only* pattern for "play a skid when a fast mover reverses or stops, but only if it was actually moving fast." Built without engine changes because no velocity is exposed to XML and the manual direction-*switch* (`HotkeyManager.setBehaviorAllSafe`‚Üí`buildBehavior`) bypasses both `buildNextBehavior` and the transition system. The design rule (Ko's): the skid must run **when appropriate and otherwise leave the normal pool untouched** ‚Äî never a forced "skid-or-X" successor (an exclusive `Add="false"` `NextBehaviorList` listing only `DashBrake`+`StandUp` was an early stop-gap and is exactly what to avoid). Pieces: **(1) Dash tracking:** the fast-move action (`Run`) stamps `mascot.setUserData('dashStamp', Date.now())` **every tick** via a side-effecting `#{...}` Animation `Condition` that returns true (must be `#{}` not `${}` so it re-evaluates per frame ‚Äî `${}` runs once at init and goes stale during a long hold; same side-effect trick as the `jumpExitVelocityX` line); the `Dash` action (the skid frame) **zeroes `dashStamp`** the same way, so a skid consumes the "recently ran" state ‚Üí fires once per stop, never loops, independent of skid duration. **(2) Stop (the main case): `DashBrake` is a normal POOL behavior** gated on `#{(Date.now()-dashStamp < 250) && <on ground>}` with a **very high `Frequency` (100000)** so it ~always wins the weighted pick when eligible and is simply *absent* otherwise (pool runs normal) ‚Äî idiomatic, same shape as `GlitchingOut`'s load-gated `Freq=1000`. This covers every stop (manual release/tap, autonomous walk-end) with **one** mechanism and **no per-behavior `NextBehaviorList`s** ‚Äî those were all deleted. Caveat: high-freq is "‚âàalways," not strictly 100%; true 100% would require gating *every other* pool behavior on `!recentlyDashing`, not worth it. **(3) Reversal:** a direction *switch* bypasses the pool (`setBehaviorAllSafe`‚Üí`buildBehavior`), so it needs its own hook ‚Äî each `MoveLeft/Right`/`Sprint*` Sequence has a `DashBrake` (= `Dash`) as its **first** child, gated on `#{<facing the OLD way> && recentlyDashing && <on ground>}`. It plays *before* the `Look` flip, so ‚Äî since `Pose` velocity moves in the facing direction (`Pose.java`: `isLookRight()?-dx:dx`) ‚Äî the `Dash` slides in the OLD direction = momentum skid, then `Look` flips and the run proceeds. (Autonomous walks never reverse ‚Äî `WalkAlongWorkAreaFloor`/`WalkAlongIECeiling` target `lookRight ? +d : -d`, always forward ‚Äî so they need no reversal hook; the pool handles their stop.) Works for any driver (hotkey, autonomous, future AI) as long as movement runs through the shared `Run`/`Move*` actions. Tunables: the `250`ms recency window and the `DashBrake` pool `Frequency`. Load-bearing `NextBehaviorList`s left intact: `ManualJump‚ÜíFall`, `HoldOntoWall‚ÜíSlideDownWall`, `SmartJump‚ÜíMove*`. Mario is not in the default showcase `ActiveShimeji`. **NOTE:** the engine's built-in `Turn="true"` flag (`Move.java`/`Animation.isTurn()`) is the *other* way to do direction-change animations, but it's unused/untested in this fork, flips facing instantly (no momentum slide), and can't express the stop case ‚Äî hence the userData-stamp approach here.
- **ClearTintOnDisable** (behavior attribute): For `Toggleable` behaviors that drive a live Tint. When unchecked ‚Üí `snapClearTint()` fires AND `setBehavior(buildNextBehavior(...))` is called to break out of the running Loop=true Sequence (otherwise `Tint.init()` re-registers the expression on the next loop iteration, un-clearing the tint). When checked (with `Frequency="0"`) ‚Üí `setBehavior()` called directly since Freq=0 won't auto-fire. Requires `ClearTintOnDisable="true"` on the `<Behavior>` tag. Tint naturally persists during drag because `Mascot.tick()` evaluates `tintColorVar`/`tintTargetVar` independently of the current behavior ‚Äî no special drag handling needed. Toggleable behaviors are excluded from the "Set Behaviour" right-click submenu and only appear as `JCheckBoxMenuItem` in "Personal Behaviours" ‚Äî this prevents activating them without the checkbox, which would set the tint without a way to clear it. Paimon's `RaveMode` uses this.
- **Per-image-set SFX volume + the popup-dismiss gotcha** (`Mascot.java`, June 2026): the right-click menu has a `Volume` `JSlider` (directly under the Tooltip toggle) persisted as `Volume.imageset.<name>` (percent, default 100; like `Sounds.imageset.*`, so it covers the whole set, not one instance). Applied at the single playback site by setting the shared clip's `MASTER_GAIN = bakedPoseDb + 20¬∑log10(fraction)` just before `start()` (0% skips the play = mute). `Sounds` tracks each clip's baked pose dB (`getBaseGain`) so the per-set volume layers on without clobbering per-pose volume; shared/cached clips are safe because the existing `!isRunning()` guard means two mascots never drive the same clip instance at once. **The gotcha (cost three wrong-layer fixes):** a `JSlider` would not stay in the `JPopupMenu` ‚Äî clicking it closed the menu ‚Äî and the cause was NOT Swing's `MenuSelectionManager`/grabber. The menu is force-closed by a **JNativeHook global OS-level mouse listener** (`popupDismissListener`, registered while the popup is open) that fires on *any* press anywhere and closes via a 150 ms timer. Fix: capture the popup's screen rect on show (`activePopupBounds`) and have `nativeMousePressed` ignore presses *inside* it (Swing already handles inside-clicks ‚Äî item activation / the slider), only force-closing on outside clicks, which is its actual purpose. **Lesson: when a popup won't stay open during interaction, check for a global native mouse hook before debugging Swing.** Belt-and-suspenders on the Swing side: the slider is wrapped in `SliderMenuItem` (a `JMenuItem` that forwards mouse events to the slider and never activates, so it can't `clearSelectedPath`), and `popup.setLightWeightPopupEnabled(false)`.

## AI Assistant Layer

All in `src/com/group_finity/mascot/assistant/`. Activates if `<Personality>` block is non-empty.

### Assistant Package Files

- **`AssistantBubble.java`** ‚Äî Floating bubble UI. `Path2D` shape, per-message fade, click-to-reply, lerp movement.
- **`AssistantInputDialog.java`** ‚Äî Typed input (`JDialog` for focus on Windows).
- **`ConsoleTap.java`** ‚Äî Global capture of the console/log stream for 2B's readout (below). `install()` (idempotent, called from `Main.run()` early) adds a root-logger `Handler` (captures the *alive* stream ‚Äî TickWatch/peer/memory/situation/drive ‚Äî at `Level.ALL`, independent of the disabled `ConsoleHandler` in `logging.properties`, so it works even though Debug.bat's real console is nearly empty) AND tees `System.out`/`System.err` (raw prints + stack traces). Rolling 200-line ring, multi-line messages collapsed to one line, 240-char cap, `version()` counter for cheap change-detection. **Filters the whisper energy-skip chatter** (`Post-NR energy ‚Ä¶ skipping Whisper`) so it never reaches the readout.

### Mascot.java AI Wiring

- **`buildSystemPrompt()`** ‚Äî Personality + memory + peer speech context + envCtx (time/location/weather) + rules.
- **`isEphemeralQuery(text)`** ‚Äî True for time, weather, timer/alarm, screen, AND factual drive lookups (`DriveIndexTool.isFactualDriveQuery` ‚Äî recency/aggregate/location intents). Gates `recordUserExchange()`. Timers included ‚Äî recording causes model to re-set reminder on every interaction. Drive lookups included ‚Äî their answer changes as files change, so recording "X is the newest file"/"124.6 GB total" entrenches a stale/wrong fact (observed June 13 2026: a wrong recency answer got stored in Paimon's memory).
- **`handleScreenQuery()`** ‚Äî Screen capture on daemon thread ‚Üí `generateWithImage()`. Appends audio context. Not recorded.
- **`fireSpontaneousComment()`** ‚Äî 45‚Äì90s cadence, reacts to active window title. Title fetch on dedicated daemon thread (prevents busy apps stalling animations). Gated by `globalSpontaneousLastFiredMs` (2 min).
- **`fireAudioReaction()`** ‚Äî 45‚Äì90s cadence. Skips if >70% word overlap with last transcript (Whisper hallucination guard). Gated by `globalAudioLastFiredMs` (90s).
- **`firePeerReaction()`** ‚Äî Triggered by `MascotSpeechRegistry`. Uses `buildLightMemoryBlock()`. **Does NOT inject `peerCtx`** (quoted peer lines act as style examples the 4B model imitates, dragging every character into one shared register ‚Äî "accommodation"; the speaker's line in the user prompt is the only context a reply needs). Includes an anti-mirroring rule: reply in your own voice, never borrow the speaker's vocabulary/phrasing/sentence shape. Filtered by `isTrivialAcknowledgement()` ‚Äî responses that normalize to "noted", "acknowledged", "understood", "affirmative", "confirmed", "copy that", "roger", or "indeed" are silently dismissed without recording. Prompt rules include: output only own words (never prefix with another character's name/dialogue tag), end with a complete sentence (no trailing ellipsis or open fragments).
- **`fireUserSpeechReaction(micText)`** ‚Äî Overheard-speech reaction (June 2026): user talks without calling a name (voice call, reacting aloud to media) ‚Üí VoiceCommandListener hands one random mascot the echo-stripped mic transcript. The mascot transcribes the 15s system-audio buffer first (the other side of the call / what the user is reacting to) and reacts to the exchange as a whole; system audio is injected as background-only context. Gates: ‚â•4 words, 2-min global cooldown, `isIncompleteUtterance` dismissal. **Re-strips the mic text against the FRESH system transcript** before reacting ‚Äî the strip VCL did used `lastSysTranscript`, which can be minutes stale (it only updates when something transcribes system audio), so newer speaker content slipped through as "user speech"; if the re-strip empties the text it was echo ‚Üí abort and roll back the cooldown. Stores `[Observed] User said (not to me): ...` fact (observation only); records to MascotSpeechRegistry at depth 0.
- **Global Reaction Cooldowns** ‚Äî Four `AtomicLong`s: spontaneous (2 min), audio (90s), vision (3 min), user-speech (2 min). Shared across all mascots; each type independent.
- **`QUICK_REACTION_STYLE_RULES`** ‚Äî Shared rules constant injected into all 4 unprompted-reaction prompts (peer, spontaneous, audio, vision): REACT-don't-report (must contain an opinion/feeling/quip about ONE detail ‚Äî bans caption-style description and "X while Y" context-splicing), never grade anyone's statement, no hedging ("appears"/"seems"/"suggests"/"indicates"/"implies" ‚Äî "say what something IS"), context lines are background-only. Evolved June 2026 across three chat.log audits: banning meta-grading alone made neutral *description* the model's fallback; banning "appears/seems" made "Your X **suggests** Y" the universal replacement template (60+ uses in one day, all four mascots) ‚Äî hedging is the model's behavior, not a vocabulary item, so the rule pairs the ban with a positive instruction. Audit note: mild tone-matching between mascots is a FEATURE per Ko ("code-switching", charming) ‚Äî only flag full register loss (a mascot emitting another's sentences verbatim-style) or content-free grading loops.
- **peerCtx scope** ‚Äî `MascotSpeechRegistry.buildContext()` is injected ONLY into `buildSystemPrompt()` (direct user replies). It is excluded from peer reactions (style imitation) AND from spontaneous/audio/vision reactions (quoted peer lines leak into user-directed comments as non-sequiturs: "You watch a YouTube build while 2B mentions Unreal Engine").
- **`buildLightMemoryBlock(peerName)`** ‚Äî for peer replies, filters out `[Observed]` facts (stale media topics bleed into unrelated peer conversations); keeps genuine user facts + peer tone.
- **`polishUtterance()`** ‚Äî Deterministic output cleanup applied in every response path: strips leading `Name:` dialogue tags, wrapping quotes (straight or curly), whitespace before punctuation.
- **`isIncompleteUtterance()`** ‚Äî True for trailing ellipsis or dangling copula ("Your confidence seems..."). Unprompted reactions failing this are silently dismissed (same path as `isTrivialAcknowledgement`); direct user replies are not gated.
- **Observed facts store the observation only** ‚Äî `addFact("[Observed] ...")` no longer appends `| Reaction: <text>`; feeding the mascot's own phrasing back via memory entrenched templates. Screen-glance facts are not stored at all (no information without the reaction).
- **`fireActionFromResponse(raw)`** ‚Äî Extracts `[ACTION:...]`, `[TIMER:...]`, `[REMEMBER:...]`. Bare `[word]` attempts behavior lookup; logs WARNING if not found.
- **`TIMER_TAG`** ‚Äî `[TIMER:VALUE:reminder]`. Integer minutes capped at 1440. Wall-clock form rolls to next day if past.
- **`say(String)`** ‚Äî Does NOT call `MascotSpeechRegistry.record()` ‚Äî XML utterances don't trigger peer reactions.

## XML Behavior/Action Authoring Notes

- `Condition`, `Duration`, velocity accept JS expressions (`mascot.anchor.x`, `environment.activeIE`, etc.).
- **Reading a tunable from `settings.properties`:** call the generic accessors on the environment from any script expression ‚Äî `mascot.environment.setting('Key', defaultNum)` (returns a double) or `mascot.environment.settingText('Key', 'defaultStr')` (returns a String). Live-read every tick (no restart), fall back to the default when the key is absent/blank/unparseable. This is the no-Java-code path for making a behavior threshold configurable (the campfire transform temps use it). Defined in `MascotEnvironment`.
- Per-mascot overrides: `img/[MascotName]/conf/`.
- `Say` usage: `<Action Name="..." Type="Embedded" Class="com.group_finity.mascot.action.Say" Text="Found one." />`
- **Per-pose ImageAnchor:** `ImagePair` cache key is path-only. `Pose.next()` calls `mascot.setRenderAnchor()` each frame.
- **Anchor-snap on surface cling:** Cling actions need `ImageAnchor` AT the surface (ceiling `48,0`, wall `16,128`). ## Known Issues / Backburner

- **Voice command latency** ‚Äî Free-form queries ~6s Whisper polling latency. Hotword‚Üíbehavior dispatch instant. Mitigate by lowering `VoicePollMs`.
- **MSI Cooler Boost (implemented June 2026)** ‚Äî Driven by the `CoolerBoost` affordance: any mascot broadcasting it (any Action carrying `Affordance="CoolerBoost"`) ramps both fans to max; none broadcasting turns it off. `Manager.checkCoolerBoostState()` checks every tick (affordances toggle without a mascot-list change, so it cannot be gated on `listChanged`); `FanController` is level-triggered with a 7s re-assert. **Turn-on debounce (June 24 2026):** blue fire must be present *continuously* for `TURN_ON_DELAY_MS` (2s) before the fans actually spin up (`triggerFanOn` starts a `pendingOnSinceMs` clock on the off‚Üíon edge; `triggerFanOff` resets it) ‚Äî so an easily-resolved temperature spike (blue gone within 2s) never engages Cooler Boost at all. Turn-*off* stays immediate. Pairs with the OllamaClient CPU-load gate: the gate prevents most spikes upstream, the debounce catches the few that slip through. Per toggle it **spawns a fresh short-lived `TempSensor.exe boost on|off` process** (new WMI connection, dies immediately) that writes EC register **0x98** bit **0x80** via the `MSI_ACPI` WMI interface ‚Äî the same firmware path MSI Center uses. The long-running TempSensor.exe is now purely the temperature streamer; fan commands do NOT go through its stdin (that pipe was a failure source ‚Äî see below). **FIXED June 13 2026; validated 1 hr / 405-of-405 toggles.** Needs admin; TempSensor is already elevated. Key mechanics learned the hard way:
  - **Use `Get_Data` / `Set_Data` (the in+out *addressed* accessors), NOT `Get_EC`/`Set_EC`.** `Get_EC` is out-only and returns a fixed 32-byte BIOS-info block (literally the BIOS version string), not addressable EC ‚Äî a `Set_EC` write goes nowhere useful. `Get_Data`/`Set_Data` package layout: `Bytes[0]=register address`, `Bytes[1]=value`.
  - **Read-modify-write is mandatory.** Reg 0x98 is `0x02` when boost is off, `0x82` when on ‚Äî a sibling bit `0x02` must be preserved, so ON = `cur | 0x80`, OFF = `cur & ~0x80`. A blind `0x80`/`0x00` write clobbers it.
  - Access uses the legacy `System.Management` `ManagementObject.InvokeMethod` API (csproj `PackageReference`, `NoWarn=CA1416;MSB3277`) ‚Äî the modern CIM cmdlets / `Invoke-CimMethod` fail on the embedded `Package_32` argument with `WBEM_E_INVALID_PARAMETER (0x80041008)`, and so would any C# mirroring them.
  - **Why per-toggle process isolation (the hard-won core):** an in-process long-lived `MSI_ACPI` WMI connection eventually **hangs on a call** (~15 min in) ‚Äî a blocking COM call never hits a GC safe-point, so the .NET GC can't run and **every managed thread freezes** (the old ~10-min full-process stall; the in-process watchdog couldn't even fire). The connection was originally cached to stop a handle leak (every `ManagementObject`/`Searcher` is an IDisposable COM wrapper), but that caching is what went stale. A long-lived Java‚ÜíTempSensor stdin command pipe also silently stopped delivering after ~20 min. Spawning a short-lived child per toggle (commit 09fb5d0) kills both failure modes; stdin survives only as a parent-death lifeline. **Lessons:** in-process WMI/COM can't be made watchdog-safe (a hang freezes the GC ‚Üí all threads) ‚Äî isolate killable work in a separate process; prefer a short-lived process per command over long-lived pipes/connections; "works fresh, dies after a while" burned through FOUR wrong root-causes before two-sided logging nailed it ‚Äî instrument both ends and read the logs, don't theorize.
  - TempSensor CLI test modes (run elevated): `boost on|off` (RMW + self-verifying read-back), `read <hexReg>`, `write <hexReg> <hexVal>`, `dump`.
